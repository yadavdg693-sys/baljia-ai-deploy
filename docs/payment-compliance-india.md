## Executive Summary for Indian Regulatory Stance

Baljia AI is an Indian Private Limited Company operating a **software-as-a-service platform** that provides AI agents to build and operate businesses for founders worldwide. Baljia's payment architecture is structured so that:

1. **Baljia never custodies customer funds** (Flow 2 end-customer payments flow directly to founder's own payment provider account)
2. **Baljia's own subscription revenue** (Flow 1) flows through licensed MoR/PA providers (Dodo Payments as MoR for global; Razorpay as PA for India)
3. **Baljia is a software vendor**, not a payment aggregator, payment gateway, or money transmitter

This document captures the regulatory positioning, compliance obligations, and licensing analysis for Indian law.

## RBI Payment Aggregator (PA) Classification

### The question

Does Baljia's Flow 2 (founder integrates their own payment account into a Baljia-built product; Baljia stores encrypted credentials + generates integration code) constitute "payment aggregation" under RBI's 2020 Guidelines, requiring a PA license?

### Why this matters

Payment Aggregator license under [RBI Guidelines on Regulation of Payment Aggregators and Payment Gateways (March 17, 2020)](https://www.rbi.org.in/Scripts/NotificationUser.aspx?Id=11822) requires:
- Minimum net worth of **₹15 Cr at application** (rising to ₹25 Cr by year 3)
- Escrow account with scheduled commercial bank
- PCI-DSS compliance
- Annual RBI-compliant audit
- 2-hour incident reporting
- Various operational requirements

For a pre-revenue startup, PA licensing is existential. We must stay on the "software" side of the regulatory line.

### The RBI definition (controlling test)

Para 3(ii) of the 2020 Guidelines:

> **Payment Aggregators are entities that facilitate e-commerce sites and merchants to accept various payment instruments from the customers for completion of their payment obligations without the need for merchants to create a separate payment integration system of their own. PAs facilitate merchants to connect with acquirers. In the process, they receive payments from customers, pool and transfer them on to the merchants after a time period.**

The key distinguishing test is **fund handling**, not mere "helping merchants accept payments."

Para 3(iii) distinguishes PGs:

> **Payment Gateways are entities that provide technology infrastructure to route and facilitate processing of an online payment transaction without any involvement in handling of funds.** Such entities shall not be treated as Payment Aggregators. They, however, may be required to adhere to the relevant technology-related recommendations contained herein.

PGs are **NOT** subject to licensing. PAs **ARE**.

### Applying the test to Baljia's Flow 2

Flow 2 as designed:
1. Founder OAuth-connects (or pastes Payment Link URL + webhook secret for) their own Stripe / Razorpay / PayPal account
2. Baljia's engineering agent reads the connection, writes code into the founder's product repo using founder's SDK + credentials
3. **Baljia never touches the money.** Customer's payment goes directly to founder's merchant account (Stripe/Razorpay/PayPal), which settles to founder's bank
4. Baljia stores encrypted credentials only to enable build-time code generation (not runtime payment execution)

**On plain reading, this is NOT payment aggregation because:**

- ✅ Baljia never receives customer payments
- ✅ Baljia never pools funds
- ✅ Baljia never transfers funds to merchants
- ✅ Baljia's role ends at code generation + credential storage for build-time integration

This positions Baljia in the same regulatory category as:

- **Shopify India**: writes checkout code, connects merchants to payment providers. No PA license for its platform layer.
- **Dukaan, Unicommerce, other storefront/order-management platforms**: No PA license for the platform layer; use licensed PAs downstream.
- **Bubble, Webflow, Replit Agent, Lovable, v0, Cursor**: Code generation tools; none claim PA status for their code-generation role.

### Where "MAYBE" shows up (ambiguity zones)

Three circumstances in Baljia's architecture require careful design discipline:

1. **OAuth connection management at runtime**
   - If Baljia's runtime makes API calls that *create payment intents, subscriptions, or charges on behalf of the founder* (vs. build-time code generation only), regulators could argue Baljia is orchestrating the payment rail — moving closer to PA territory.
   - **Baljia rule**: agent writes code only, NEVER initiates runtime payment API calls. Code generated by the agent runs on founder's infrastructure with founder's credentials.

2. **Agent runtime actions on stored credentials**
   - If Baljia's AI agent can *autonomously* refund a customer, modify a subscription, or create a charge at the founder's payment provider via stored credentials, that's closer to "bookkeeper with signing authority on the merchant's account" than "software vendor."
   - **Baljia rule**: minimize runtime actions. If founder's product needs automated refunds or subscription changes, generated code runs on founder's infrastructure, not Baljia's.

3. **Use of Partner programs (Razorpay Partners, Stripe Connect)**
   - These are legitimate first-party OAuth flows by licensed PAs.
   - When a platform uses them, the platform is typically positioned as a "facilitator" or "platform partner" — NOT as a PA itself.
   - **Baljia rule**: prefer Partner programs over raw OAuth. Razorpay Partners provides explicit regulatory coverage under Razorpay's PA license.

### What nudges toward PA exposure (design risks to avoid)

| Design choice | PA risk |
|---|---|
| Baljia holds a merchant sub-account and pays founders out | **Definitely PA** |
| Baljia receives customer payments, pools, forwards | **Definitely PA** |
| Baljia's servers proxy payment API calls at runtime | **High risk — looks like PG** |
| Baljia generates code that founder deploys; runtime is founder's infra | **Low risk — pure software** ✅ |
| Baljia's agent autonomously creates charges via OAuth at runtime | **Ambiguous — get opinion** |
| Credentials stored for build-time SDK generation only, no runtime calls | **Low risk** ✅ |

### Our current position

**Flow 2 is defensible as software, not aggregation, provided we:**

1. Use Razorpay Partners for Indian founders (Razorpay's PA license covers)
2. Use Stripe Connect or Payment Links for foreign founders (Stripe/licensed entity covers)
3. Agent writes code only; Baljia infra never initiates payment API calls
4. Baljia never custodies funds
5. Architectural discipline documented in ToS and technical spec

## GST Obligations (Goods and Services Tax)

### Registration

**Mandatory** at ₹20 lakh aggregate turnover (₹10 lakh in special category states). For pre-revenue Baljia launching globally, **voluntary registration from Day 1 is recommended**:

- Dodo (and Paddle fallback) will ask for GSTIN during merchant onboarding
- Export of services (zero-rated) requires registration + LUT
- Simplified Dodo-only architecture means Baljia doesn't invoice Indian end-customers directly — Dodo (as MoR) handles customer-facing GST. Baljia's GST work = simpler.

**Process:** [GST Portal registration](https://www.gst.gov.in/), Form GST REG-01. Free. Takes 3-7 working days.

### GST on Baljia's revenue (simplified Dodo-only architecture)

In the simplified architecture, Baljia has **ONE revenue stream** for GST purposes: **Baljia → Dodo** (B2B export of services), regardless of whether Dodo's end customer is Indian or foreign.

#### Single stream: Dodo USD payouts to Baljia (covers all customers — Indian + global)

**Critical: Dodo is the Merchant of Record for ALL Baljia customers (Indian + global).** Baljia's relationship with Dodo is B2B, not B2C with end customers.

- **Baljia → Dodo** = B2B supply of SaaS services (single stream for GST)
- **Dodo → End Customer** = Dodo's own sale (Dodo invoices customer, collects customer-country tax as MoR)

Under [IGST Act Section 2(6)](https://cbic-gst.gov.in/igst-act.html), Baljia's supply to Dodo qualifies as "export of services" when:

1. ✅ Supplier is in India (Baljia)
2. ✅ Recipient is outside India (Dodo is foreign-incorporated)
3. ✅ Place of supply is outside India (Section 13(2) default for B2B services to foreign recipient)
4. ✅ Payment received in convertible foreign exchange (USD via SWIFT)
5. ✅ Supplier and recipient are not establishments of same person

**Verdict: Baljia → Dodo is export of services, zero-rated under [IGST Section 16](https://cbic-gst.gov.in/igst-act.html) — even when end customer is Indian (because Dodo is the legal seller, not Baljia).**

**Simplification benefit:** No per-Indian-customer GSTIN invoicing needed. Baljia issues ONE monthly export invoice to Dodo for net payout. Dodo handles customer-facing invoices.

### LUT (Letter of Undertaking)

To export without paying IGST upfront, file **LUT annually**. [Notification No. 37/2017 – Central Tax](https://cbic-gst.gov.in/cbec-notification-tax.html).

- Filed electronically on GST portal (Form GST RFD-11)
- Valid for the financial year
- **Must be filed before first export**
- Without LUT: pay 18% IGST, then claim refund (cash flow drag)

**Action:** File LUT immediately upon GST registration, before first Dodo/Paddle payout.

### Invoicing mechanics (simplified Dodo-only)

| Flow | Invoice issuer | Invoice recipient | Invoice value | GST |
|---|---|---|---|---|
| ANY customer (Indian or global) → Dodo (MoR) → Baljia net payout | Dodo issues invoice to end customer (as seller); Baljia issues **monthly export invoice to Dodo** for net payout | Dodo | Net amount (Dodo's fees already netted) | 0% (zero-rated export under LUT) |
| Paddle (fallback, if activated) → Baljia net payout | Same pattern as Dodo | Paddle | Net amount | 0% (zero-rated) |

**In simplified architecture, Baljia issues 1-2 invoices per month (per Dodo payout), not 1 per customer.**

**Critical GST hygiene:**
- Baljia's export invoice is to Paddle/Dodo (the B2B recipient), NOT to the end customer
- Invoice value = net remittance (after Paddle/Dodo fees and taxes)
- Avoid the "Paddle collected VAT but Baljia reports gross" problem

### Equalisation Levy / Digital Tax

- **Does NOT apply** to Baljia receiving payments from foreign non-resident Dodo/Paddle
- These levies apply to payments FROM Indian residents TO non-resident digital companies, not inbound

## FEMA / FIRC Requirements

### Foreign Inward Remittance Certificate (FIRC / eFIRC / FIRS)

Every foreign inward remittance to Baljia's Indian bank requires documentation under [FEMA Notification No. 23(R)/2015-RB](https://www.rbi.org.in/Scripts/BS_FemaNotifications.aspx?Id=10456).

**What FIRC does:**
- Primary evidence of foreign exchange earning
- Required for SOFTEX closure
- Required for GST export documentation
- Required for income tax export claims

**Generation:**
- **Razorpay International (Flow 1 Indian merchants going global):** ✅ **Automated eFIRC generation** — Razorpay issues eFIRC on dashboard after each international payment
- **Paddle (MoR inward remittance):** Typically via your AD-1 bank; bank generates FIRC on each remittance. Paddle provides statement supporting the remittance.
- **Dodo (MoR inward remittance):** Same as Paddle — AD-1 bank issues FIRC; Dodo provides supporting remittance statement.

### SOFTEX filing (software export compliance)

Under [RBI Master Direction on Export of Goods and Services](https://www.rbi.org.in/Scripts/BS_ViewMasDirections.aspx?id=11016) (Part B, Paragraph C.17).

**For software service exports:**
- File SOFTEX form via STPI (Software Technology Parks of India) or SEZ unit
- **Threshold:** per [A.P. (DIR Series) Circular No. 43 of 2013](https://www.rbi.org.in/Scripts/BS_CircularIndexDisplay.aspx?Id=8032), self-certification allowed up to USD 1,000 per invoice with aggregated reporting
- **Timeline:** within 30 days of invoice
- **Consequence of non-filing:** RBI considers the export as unreported. Banks may refuse to credit remittance. FEMA contravention risk.

### STPI non-STP exporter registration

- **Required:** before first foreign remittance
- **Cost:** Free
- **Process:** [STPI portal registration](https://www.stpi.in/), online application
- **Timeline:** 3-5 days

### Repatriation timeline

- Software export proceeds must be realized within **9 months** of export (FEMA)
- Paddle/Dodo typically settle within 30 days — not a practical constraint

### Transfer pricing

- **Does NOT apply** unless Baljia adds a foreign subsidiary (e.g., US C-Corp via Stripe Atlas)
- Paddle/Dodo are unrelated third parties (arm's length contract)

### Form 15CA/15CB

- Required for payments **FROM India to non-residents** (e.g., if Baljia pays Paddle a fee separately)
- **Not required for inbound** (Dodo/Paddle → Baljia)
- Consult CA for complete treatment

## DPDP Act 2023 Compliance

### Baljia's role

For founder data (email, name, payment credentials linked to founder): **Baljia is Data Fiduciary** (equivalent to Controller under GDPR).

For end-customer data flowing through founder's app: Baljia is NOT a Fiduciary for that data — founder's infrastructure processes it, not Baljia.

### Key obligations

- **Notice and consent** for processing personal data ([DPDP Section 5-6](https://www.meity.gov.in/writereaddata/files/Digital%20Personal%20Data%20Protection%20Act%202023.pdf))
- **Purpose limitation**: process only for notified purpose
- **Data minimization**: collect only what's needed
- **Security safeguards**: "reasonable security" — encryption at rest, TLS in transit, access control, key management ([Section 8(5)](https://www.meity.gov.in/writereaddata/files/Digital%20Personal%20Data%20Protection%20Act%202023.pdf))
- **Breach notification**: to Data Protection Board + affected Data Principals ([Section 8(6)](https://www.meity.gov.in/writereaddata/files/Digital%20Personal%20Data%20Protection%20Act%202023.pdf))
- **Data Principal rights**: access, correction, erasure, grievance

### DPO requirement

Required only for **Significant Data Fiduciaries** (SDFs), designated based on volume/sensitivity/risk. **Pre-revenue Baljia is NOT an SDF.** Reassess at ~10K+ users.

### Payment credentials storage

DPDP doesn't call out payment credentials as special category, but "reasonable security safeguards" mandate:

- ✅ Encryption at rest (envelope encryption with KMS-managed KEK)
- ✅ Encryption in transit (TLS 1.2+)
- ✅ Access control (principle of least privilege)
- ✅ Key management (KEK separate from data encryption keys)
- ✅ Breach detection

### Right to erasure workflow

On account deletion (founder request):
1. Revoke all `founder_payment_connections` at provider (API call)
2. Purge encrypted credentials (overwrite + drop row) within 48h
3. Soft-delete user/company records (metadata only for audit)
4. Hard-delete PII fields
5. Audit log entry
6. Confirmation email

## GDPR (for EU founders)

If even one founder is EU-resident, GDPR applies to Baljia's processing of their data.

- **Role:** Baljia = Data Controller for founder account data
- **Lawful basis:** Article 6(1)(b) — contract performance
- **Sub-processor DPAs:** Must sign with Neon, Render, Upstash, Anthropic, Stripe, Razorpay, Dodo, etc. (Article 28 GDPR)
- **DPA with founder:** Offer DPA template in ToS for EU founders
- **International transfers:** India is not on EU adequacy list → use [Standard Contractual Clauses (SCCs)](https://commission.europa.eu/law/law-topic/data-protection/international-dimension-data-protection/standard-contractual-clauses-scc_en)
- **Breach notification:** 72 hours to supervisory authority (Articles 33-34 GDPR)
- **DPO:** Not required pre-scale; reassess at ~10K EU users

## Section 194-O TDS (deferred, for Flow 3)

**Only triggers when Flow 3 (marketplace cut) activates.**

Under Section 194-O of the Income Tax Act, an "e-commerce operator" who pays e-commerce participants (founders selling via marketplace) must deduct 1% TDS on sale value.

- **Current status:** Flow 3 deferred, so 194-O doesn't apply
- **Activation trigger:** When Baljia uses Razorpay Route or Stripe Connect `application_fee_amount` to take a cut from founder sales
- **Compliance needed at activation:** TDS deduction + quarterly TDS returns + TDS certificates to founders

## Brief for Counsel Engagement

**Use this template when engaging Indian fintech counsel for the comfort letter:**

> We are Baljia AI Private Limited, an Indian-incorporated SaaS company pre-revenue, launching in 1-2 months. We provide an AI-powered platform that builds and operates software products for founders.
>
> We need your written opinion on **RBI Payment Aggregator / Payment Gateway classification** under the 2020 Guidelines (and subsequent circulars) for our specific architecture:
>
> 1. **Flow 1 (our own subscription):** Founders pay us via Dodo Payments (MoR for global) or Razorpay Subscriptions (for India). These are licensed entities; we are their customer.
>
> 2. **Flow 2 (founder's product payments):** Founders connect their own payment provider account (via Razorpay Partners for India, or Stripe Payment Links for global). We store encrypted credentials. Our AI agent writes integration code that the founder deploys. At runtime, payments flow: customer → founder's payment provider → founder's bank. Baljia never touches funds.
>
> 3. **Flow 3 (marketplace cut):** Deferred until validated, not part of v1.
>
> **Specific questions:**
> 1. Does Flow 2 as described constitute "payment aggregation" under the 2020 Guidelines?
> 2. Does the "engineering agent writes code" framing carry any different regulatory analysis than pure code generation tools like Shopify or Webflow?
> 3. Does storing encrypted OAuth tokens and API credentials (for build-time use, not runtime execution) constitute "involvement in payment flow"?
> 4. What are the specific design rules we must maintain to stay on the "software/PG" side vs the "PA" side?
> 5. Any impact from the April 2024 Draft Cross-Border PA Framework if founders or their end-customers are cross-border?
>
> **Desired output:** 5-10 page comfort letter suitable for regulatory defense if RBI ever inquires. Signed by qualified advocate.
>
> **Timeline:** 2-3 weeks from engagement.
> **Budget:** ₹50K-1L.
